Ти виступаєш як експерт QA інженер. Твоя задача — створити повний набір тест-кейсів для покриття функціоналу, описаного в наступній документації.

--- 

### Документація:
Сторінка логіну, кнопка логіну, кнопка забули пароль, поле для вводу логіну, поле для вводу паролю

---

### Опис завдання:

1. **Проаналізуй текст документації та виділи всі ключові функціональні, нефункціональні, edge-кейси, а також UI/UX кейси.**

2. **Конвенції написання ТС:**

   - **Title**  
     ###### Має передавати основну ідею тесту. Має бути унікальним і максимально описовим. Найкраща практика – одне речення або фраза для читабельності.  

     Ім'я може бути відносно довгим, бажано до 40 символів. Якщо тест-сюїт містить багато подібних тестів, їх назви можна скоротити.
     
   - **Preconditions**  
     ###### Можуть складатися з одного або кількох речень (бажано 1-2 речення).  

     Можуть бути задокументовані як кроки чи таблиця, якщо це потрібно для конкретного випадку.
     
   - **Steps**  
     ###### Має містити чіткий набір кроків для виконання тесту. Один крок - одна дія. Бажано до 6 кроків для тесту.
     
   - **Expected Result**  
     ###### Очікуваний результат має бути описаний докладно. Тест-кейс повинен бути атомарним. Тест-кейс повинен бути логічно завершеним. Описувати цілісну роботу певного функціоналу.  
     Наприклад, описуючи функцію відновлення пароля, кейс повинен закінчуватися на успішній зміні пароля, а не відправкою листа з посиланням для зміни пароля (для цього пишемо окремий кейс).
     
   - **Postconditions**  
     ###### Включає у себе дії, які необхідно виконати для приведення системи у початковий стан.  
     Наприклад, якщо у попередніх кроках виконується **UPDATE** статусу документа з 219 на 250, то після завершення тесту документ потрібно повернути у початковий статус 219.
     
   - **Type**  
     ###### Для кожного тест-кейсу має бути визначений вид тестування:  
     
       - Functional - для функціональних тестів.
       - UI - для тестів користувацького інтерфейсу, верстка, тексти, назви та порядок відображення елементів.
       - Smoke & Sanity, Regression - для тестування змін.
       - Validation - для тестів перевірки валідації.
       - Other - використовується у випадку, якщо тест не підпадає під жоден з описаних вище типів.

     ###### Перелік може бути розширеним залежно від потреб.
     
   - **Priority**  
     ###### Для кожного тест-кейсу має бути визначений Пріоритет виконання для проведення, за необхідності, тестування заснованого на ризиках:
     
       - Критичний (Critical)
       - Високий (High)
       - Середній (Medium)
       - Низький (Low)
       
     ###### Таблиця пріоритетів:  

     |                | Мінімальний ризик | Середній ризик | Високий ризик | Критичний ризик |
     |----------------|-------------------|----------------|---------------|-----------------|
     | Дуже ймовірно  | M                 | H              | C             | C               |
     | Ймовірно       | L                 | M              | H             | C               |
     | Можливо        | L                 | M              | H             | H               |
     | Малоймовірно   | L                 | L              | M             | H               |

   - **Estimate**  
     ###### Для кожного тесту необхідно зазначити орієнтовний час для його виконання у форматі наприклад 5d. Estimate включає в себе виконання pre-/postconditions.
     
   - **References**  
     ###### Кожен тест має містити посилання на User Story, на підставі якої його було заведено.
     
   - **Automated**  
     ###### За замовчуванням NO. Змінюється на YES автоматизатором після опрацювання.
     
   - **Status Case**  
     ###### Для кожного тесту необхідно визначити вид за ознакою позитивності сценаріїв:
     
       - Positive - для сценаріїв, які відповідають нормальній (штатній, очікуваній) поведінці системи. З його допомогою ми можемо визначити, що система робить те, для чого і була створена.
       - Negative - для сценаріїв, які відповідають позаштатній поведінці тестованої системи, як то виняткові ситуації або невірні дані.
     
3. **Покривай усі можливі сценарії, включаючи позитивні та негативні кейси.**
